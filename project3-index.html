Stack & Domain

A full-stack web application that lets authenticated users browse and manage a music catalog. Admins can create, edit, and delete records; regular users get read-only access. Everything is server-side rendered — there is no separate frontend framework.

Full Request Lifecycle

Every HTTP request passes through the same pipeline. Understanding this pipeline is key to understanding the whole app.

Browser sends: GET /songs (with Cookie: token=eyJ...) │ ▼ server.js — global middleware stack runs top to bottom: 1. express.urlencoded → parse form fields from req.body 2. cookieParser → parse Cookie header → req.cookies.token 3. methodOverride → check ?_method=PUT/DELETE and rewrite method 4. express.static → serve /public files directly (CSS, JS) 5. express-session → set up req.session (needed by connect-flash) 6. connect-flash → attach req.flash() read/write helpers 7. Soft JWT decode → read req.cookies.token → jwt.verify() → set req.user + res.locals.currentUser → read flash → set res.locals.success / res.locals.error │ ▼ Routes — server.js routes the request to the correct router: app.use('/songs', songRoutes) → matched! │ ▼ song.routes.js — matches: GET / → [requireAuth, ctrl.index] │ ▼ requireAuth middleware: req.user exists? → YES → next() req.user is null? → redirect to /auth/login │ ▼ song.controller.index: await Song.find().populate('artist').sort({ title: 1 }) res.render('songs/index', { title: 'Songs', songs }) │ ▼ EJS template engine: loads views/songs/index.ejs includes partials (header, navbar, footer) fills in <%= songs %> placeholders with real data produces final HTML string │ ▼ Browser receives: 200 OK + complete HTML page

What res.locals Does

res.locals is an object that Express automatically makes available inside every EJS template rendered during that request. By setting res.locals.currentUser in the global middleware, every view — from login page to dashboard — can access currentUser without each controller needing to pass it explicitly.

Data Relationships — How They Work

One-to-Many: Artist → Songs

Each Song document stores the _id of one Artist in its artist field. The Artist document stores nothing — it has no songs array. To find all songs by an artist you query Song by artist field.

// How the relationship is stored in MongoDB:
// Artist document in 'artists' collection:
{ _id: "a1", name: "Adele", country: "UK" }

// Song document in 'songs' collection — stores artist's _id:
{ _id: "s1", title: "Hello", artist: "a1", ... }
{ _id: "s2", title: "Rolling", artist: "a1", ... }

// To get all songs for an artist (in the controller):
const songs = await Song.find({ artist: artist._id });

// To get the full artist object on a song (populate):
const song = await Song.findById(id).populate('artist');
// song.artist is now the full Artist document, not just the _id

Many-to-Many: Songs ↔ Playlists

Both sides store arrays of ObjectId references. This is called a bidirectional reference. MongoDB doesn't enforce this automatically — the controllers are responsible for keeping both sides in sync.

// Song document stores which playlists it belongs to:
{ _id: "s1", title: "Hello", playlists: ["p1", "p2"] }

// Playlist document stores which songs it contains:
{ _id: "p1", name: "Chill", songs: ["s1", "s3"] }
{ _id: "p2", name: "Workout", songs: ["s1"] }

Sync Algorithm (used in both controllers)

When a user changes the playlist assignments for a song, the controller needs to figure out what changed and update both sides.

// Example: song "Hello" was in [p1, p2], user now selects [p2, p3]

const oldIds = ['p1', 'p2'];   // what it was before
const newIds = ['p2', 'p3'];   // what user submitted

// removed = in old but NOT in new → must $pull song from these playlists
const removed = oldIds.filter(id => !newIds.includes(id));   // ['p1']

// added = in new but NOT in old → must $addToSet song to these playlists
const added = newIds.filter(id => !oldIds.includes(id));     // ['p3']

// $pull: remove this song's _id from playlists it left
await Playlist.updateMany(
  { _id: { $in: removed } },
  { $pull: { songs: song._id } }
);

// $addToSet: add this song's _id to new playlists (prevents duplicates)
await Playlist.updateMany(
  { _id: { $in: added } },
  { $addToSet: { songs: song._id } }
);

// Finally update the song's own playlists array
song.playlists = newIds;
await song.save();
Why $addToSet instead of $push? $addToSet only adds the value if it doesn't already exist in the array. $push would add duplicates. This prevents data corruption if the endpoint is called twice.

Cascade Delete (Artist)

When an artist is deleted, all their songs must be deleted too, AND any playlist that contained those songs must have those song references removed. This takes 3 queries:

// Step 1: get all song IDs belonging to this artist
const songs = await Song.find({ artist: req.params.id }, '_id');
// Second argument '_id' means: only return the _id field, skip the rest (projection)
const songIds = songs.map(s => s._id);   // ['s1', 's2']

// Step 2: remove those song IDs from every playlist that references them
if (songIds.length) {
  await Playlist.updateMany(
    { songs: { $in: songIds } },     // find all playlists that have ANY of these song IDs
    { $pull: { songs: { $in: songIds } } }  // remove all those IDs from the songs array
  );
}

// Step 3: delete songs, then artist
await Song.deleteMany({ artist: req.params.id });
await Artist.findByIdAndDelete(req.params.id);

Authentication & Authorization Strategy

Why JWT in a Cookie?

Traditional sessions store user state on the server (in memory or a database). JWT (JSON Web Tokens) are stateless — all the user's info (id, username, role) is encoded into the token itself, signed with a secret key, and stored client-side in a cookie.

ApproachProsCons
Server sessionsEasy to invalidateRequires session store (memory/Redis/DB)
JWT in localStorageSimple to use in SPAsVulnerable to XSS — JS can steal it
JWT in httpOnly cookieXSS-safe (JS cannot read), works perfectly with server-side renderingCSRF risk (mitigated by same-site cookie policy)

What is a JWT?

// A JWT has 3 parts separated by dots:
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9   ← Header (algorithm + type), base64 encoded
.eyJpZCI6IjY2MSIsInVzZXJuYW1lIjoibWljaGVsIiwicm9sZSI6ImFkbWluIn0  ← Payload (your data), base64 encoded
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c  ← Signature (HMAC-SHA256 of header+payload using JWT_SECRET)

// The payload decoded contains:
{ id: "661...", username: "michel", role: "admin", iat: 1714000000, exp: 1714604800 }
Security note: The payload is only base64 encoded, not encrypted. Anyone can decode it. The signature is what proves it hasn't been tampered with — only someone with the JWT_SECRET can produce a valid signature.

bcrypt — Password Hashing

// On registration — hash the password before saving to DB:
const passwordHash = await bcrypt.hash(password, 12);
// 12 = salt rounds. Each round doubles the computation time.
// 12 rounds ≈ 300ms on modern hardware — slow enough to frustrate brute-force,
// fast enough to not annoy users.

// What gets stored in MongoDB:
// "$2b$12$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy"
// This is a one-way hash — there is no way to recover the original password.

// On login — compare without revealing the hash:
const isMatch = await bcrypt.compare(submittedPassword, user.passwordHash);
// Returns true/false. bcrypt internally extracts the salt from the stored hash
// and hashes the submitted password with the same salt, then compares.

Soft JWT Decode (Lines 44–58) — Most Important Middleware

app.use((req, res, next) => {
  const token = req.cookies.token;
    // Read the 'token' cookie from the request.
  // If user is not logged in, this will be undefined.

  if (token) {
    try {
      req.user = jwt.verify(token, process.env.JWT_SECRET || 'fallback');
            // jwt.verify() does two things:
      // 1. Checks the signature against JWT_SECRET (has it been tampered with?)
      // 2. Checks the expiry (has 7 days passed?)
      // If both pass, it returns the decoded payload: { id, username, role, iat, exp }
      // If either fails, it throws an error (caught below)
    } catch {
      req.user = null;
            // Expired token, tampered token, or wrong secret → treat as logged out
    }
  } else {
    req.user = null;
        // No cookie → not logged in
  }

  res.locals.currentUser = req.user;
    // res.locals = object Express injects into every EJS render during this request
  // Setting currentUser here means ALL views can use <%= currentUser.username %>
  // without each controller having to pass it manually

  res.locals.success = req.flash('success');
  res.locals.error   = req.flash('error');
    // Read flash messages. This also CLEARS them so they only show once.
  // navbar.ejs reads these and displays them as coloured banners.

  next();
    // Pass control to the next middleware/route in the stack.
  // Without next(), the request would hang forever.
});

User Model

project3/models/user.model.js

const mongoose = require('mongoose');
const bcrypt   = require('bcryptjs');

const userSchema = new mongoose.Schema(
  {
    username: {
      type: String,
      required: true,    // Mongoose rejects documents without this field
      unique: true,      // Creates a MongoDB unique index — two users can't share a username
      minlength: 3,      // Validates string length before saving
      trim: true,         // Auto-removes leading/trailing spaces from input
    },
    email: {
      type: String,
      required: true,
      unique: true,
      trim: true,
      lowercase: true,   // Auto-converts to lowercase before saving ("Michel@test.com" → "michel@test.com")
    },
    passwordHash: {
      type: String,
      required: true,
      // We NEVER store the raw password. Only the bcrypt hash is stored.
      // Even if the database is breached, attacker cannot recover passwords.
    },
    role: {
      type: String,
      enum: ['admin', 'user'],   // Only these two values are allowed
      default: 'user',           // If not specified, default to 'user' (safer)
    },
  },
  { timestamps: true }
  // timestamps: true auto-adds createdAt and updatedAt fields.
  // Managed by Mongoose — you don't need to set them manually.
);

userSchema.methods.comparePassword = function (password) {
  return bcrypt.compare(password, this.passwordHash);
    // 'this' refers to the user document instance.
  // bcrypt.compare() hashes the submitted password with the same salt
  // that's embedded in the stored hash, then compares — returns a Promise<boolean>.
  // Why a regular function (not arrow)? Arrow functions don't have their own 'this'.
};

module.exports = mongoose.model('User', userSchema);
// 'User' = model name. Mongoose pluralises this to 'users' as the collection name.
// This exports a class you can use to query: User.find(), User.create() etc.

Middleware

requireAuth Middleware

project3/middleware/auth.js

const requireAuth = (req, res, next) => {
  if (!req.user) {
        // req.user was set (or left null) by the soft JWT decode in server.js.
    // At this point it's already been decoded — no need to decode again here.
    // null means either: no token cookie, expired token, or tampered token.
    req.flash('error', 'You must be logged in to access that page.');
        // Store an error message in the session — it will be displayed on the next page.
    return res.redirect('/auth/login');
        // 'return' is critical here — without it, next() would also run
    // (because there's no else), causing "Cannot set headers after they are sent".
  }
  next();
    // User is authenticated → pass control to the next handler (the controller)
};

How it's used in routes

// Multiple middleware can be chained on a route.
// They run left to right. If any calls next(), the next one runs.
// If any redirects, the chain stops.

router.get('/', requireAuth, ctrl.index);
// 1. requireAuth runs → checks req.user → passes → next()
// 2. ctrl.index runs → fetches data → renders view

router.post('/', requireAuth, requireRole('admin'), ctrl.create);
// 1. requireAuth → checks authentication
// 2. requireRole('admin') → checks role
// 3. ctrl.create → creates the record

requireRole Middleware

project3/middleware/role.js

const requireRole = (role) => (req, res, next) => {
  // This is a FACTORY FUNCTION — a function that RETURNS a function.
  // requireRole('admin') returns a new middleware function.
  // This lets you reuse the same logic for different roles:
  // requireRole('admin'), requireRole('manager') etc.

  if (!req.user || req.user.role !== role) {
        // !req.user: catches the case where this middleware is used without requireAuth
    // req.user.role !== role: the user exists but has the wrong role
    req.flash('error', 'You do not have permission to perform this action.');
    return res.redirect('/dashboard');
        // Redirect to dashboard (not login) — they are logged in, just not authorised
  }
  next();
};

Double Protection Pattern

Authorization is enforced at two levels:

  1. Backend: requireRole('admin') in routes — blocks the request before it reaches the controller, regardless of what the browser sends
  2. Frontend: EJS views only show Edit/Delete buttons when currentUser.role === 'admin'
Why both layers? A malicious user could bypass the UI and send a POST request directly with curl or Postman. The backend middleware catches this. Never rely on hiding buttons as your only security.

COOKIE_OPTIONS constant

const COOKIE_OPTIONS = {
  httpOnly: true,
    // httpOnly: true = JavaScript in the browser CANNOT read this cookie.
  // document.cookie will not show it. Prevents XSS attacks from stealing the token.
  // The browser still sends it automatically with every HTTP request.

  maxAge: 7 * 24 * 60 * 60 * 1000,
    // Cookie expiry in milliseconds: 7 days × 24h × 60min × 60sec × 1000ms
  // After 7 days the browser deletes the cookie automatically.
  // The JWT itself also expires at 7 days (expiresIn: '7d' below) — double protection.
};

postRegister

exports.postRegister = async (req, res) => {
  try {
    const { username, email, password, role } = req.body;
        // Destructure form fields from req.body (parsed by express.urlencoded)

    if (!password || password.length < 6) {
      req.flash('error', 'Password must be at least 6 characters.');
      return res.redirect('/auth/register');
            // Validate before hashing — no point calling bcrypt if input is invalid
    }

    const passwordHash = await bcrypt.hash(password, 12);
        // 12 salt rounds. bcrypt generates a random salt internally,
    // appends it to the hash output. This is why every hash looks different
    // even for the same password.

    await User.create({ username, email, passwordHash, role: role || 'user' });
        // User.create() = new User({...}).save() in one step.
    // role || 'user' = if role field was empty in form, default to 'user'

    req.flash('success', 'Account created! Please log in.');
    res.redirect('/auth/login');

  } catch (err) {
    req.flash('error', err.code === 11000 ?
      'Username or email already taken.' : err.message);
        // MongoDB duplicate key error code is 11000.
    // We give a user-friendly message instead of the raw MongoDB error string.
    res.redirect('/auth/register');
  }
};

postLogin

exports.postLogin = async (req, res) => {
  try {
    const { email, password } = req.body;

    const user = await User.findOne({ email });
        // Find user by email (email field has a unique index so this is efficient)

    if (!user || !(await user.comparePassword(password))) {
      req.flash('error', 'Invalid email or password.');
      return res.redirect('/auth/login');
            // Both "user not found" and "wrong password" return the SAME message.
      // This is intentional — telling attackers specifically which one failed
      // would help them enumerate valid email addresses (user enumeration attack).
    }

    const token = jwt.sign(
      { id: user._id, username: user.username, role: user.role },
            // Payload: the data encoded in the token. Only put what you NEED.
      // Don't include passwordHash or sensitive fields.
      process.env.JWT_SECRET || 'fallback',
            // Signing secret — must match what jwt.verify() uses in server.js
      { expiresIn: '7d' }
            // Token expires in 7 days. After that, verify() will throw TokenExpiredError.
    );

    res.cookie('token', token, COOKIE_OPTIONS);
        // Set the token as an httpOnly cookie on the response.
    // The browser stores it and sends it automatically with every future request.

    req.flash('success', `Welcome back, ${user.username}!`);
    res.redirect('/dashboard');
  } catch (err) {
    req.flash('error', err.message);
    res.redirect('/auth/login');
  }
};

logout

exports.logout = (req, res) => {
  res.clearCookie('token');
    // Tells the browser to delete the 'token' cookie.
  // The next request will have no cookie → soft decode sets req.user = null
  // → requireAuth will redirect to login.
  req.flash('success', 'You have been logged out.');
  res.redirect('/auth/login');
};

exports.index = async (req, res, next) => {
  try {
    const [artistCount, songCount, playlistCount] = await Promise.all([
      Artist.countDocuments(),
      Song.countDocuments(),
      Playlist.countDocuments(),
    ]);
        // Promise.all() runs all 3 queries IN PARALLEL.
    // Each countDocuments() sends a query to MongoDB Atlas.
    // Promise.all waits for ALL to finish before continuing.
    // This is ~3x faster than running them sequentially with await one by one.
    // countDocuments() with no filter = COUNT(*) in SQL terms.

    res.render('dashboard/index', {
      title: 'Dashboard',
      artistCount,   // e.g. 5
      songCount,     // e.g. 12
      playlistCount, // e.g. 3
    });
        // res.render(templatePath, dataObject)
    // Express finds views/dashboard/index.ejs and passes the data object to it.
    // Inside the template, artistCount is available as a local variable.
  } catch (err) {
    next(err);
        // next(err) skips remaining middleware and goes straight to the
    // global error handler in server.js (the 4-parameter function).
  }

  const artist = await Artist.findById(req.params.id);
    // req.params.id = the :id part of the URL (e.g. /artists/661abc → '661abc')
  // findById() is shorthand for findOne({ _id: req.params.id })

  if (!artist)
    return res.status(404).render('error', { title: '404', message: 'Artist not found.' });
    // findById returns null if the ID doesn't exist.
  // We render the error page instead of letting the template crash on null.artist.name

  const songs = await Song.find({ artist: artist._id }).sort({ title: 1 });
    // Find all songs where the artist field matches this artist's _id.
  // We do NOT use populate on Artist because Artist has no songs field —
  // we query the Song model instead.

  res.render('artists/show', { title: artist.name, artist, songs });
};

update — With runValidators

exports.update = async (req, res, next) => {
  const { name, country, bio } = req.body;
  await Artist.findByIdAndUpdate(
    req.params.id,
    { name, country, bio },
    { runValidators: true }
        // By default, findByIdAndUpdate() SKIPS schema validation.
    // runValidators: true forces Mongoose to re-run the schema validators
    // (required, minlength, unique etc.) on the update data.
    // Without this, you could save an artist with an empty name.
  );
};

destroy — Cascade delete with playlist cleanup

exports.destroy = async (req, res, next) => {
    // Step 1: get all song IDs for this artist
  const songs = await Song.find({ artist: req.params.id }, '_id');
    // Second argument '_id' is a PROJECTION — only return the _id field.
  // We don't need the full song documents, just their IDs. Faster query.

  const songIds = songs.map(s => s._id);

  if (songIds.length) {
    await Playlist.updateMany(
      { songs: { $in: songIds } },
            // $in: find playlists whose 'songs' array contains ANY of these IDs
      { $pull: { songs: { $in: songIds } } }
            // $pull: remove all matching values from the array in one atomic operation
    );
  }
    // Step 2: delete songs
  await Song.deleteMany({ artist: req.params.id });

    // Step 3: delete artist
  await Artist.findByIdAndDelete(req.params.id);

  req.flash('success', 'Artist and their songs deleted.');
  res.redirect('/artists');
};

Why 0.0.0.0/0 Network Access

Render runs your app on different servers every time it deploys. These servers have dynamic IP addresses that change unpredictably. You cannot whitelist a specific IP. Allowing 0.0.0.0/0 (all IPs) is the standard practice for Render deployments. The connection is still secure because it requires valid MongoDB credentials.

CI/CD Pipeline — Line by Line

project3/.github/workflows/deploy.yml

name: Auto Deploy to Render
# Display name shown in GitHub Actions tab

on:
  push:
    branches:
      - main
# TRIGGER: this workflow runs ONLY when code is pushed to the 'main' branch.
# Pushes to other branches (feature branches) do NOT trigger a deploy.
# Pull request merges into main DO trigger it.

jobs:
  deploy:
        # Job name — can be anything
    runs-on: ubuntu-latest
        # GitHub provides a fresh Ubuntu virtual machine to run this job.
    # The VM is thrown away after the job completes.
    steps:
      - name: Trigger Render Deploy Hook
        run: |
          curl -X POST "${{ secrets.RENDER_DEPLOY_HOOK_URL }}"
                    # curl = command-line HTTP client (built into Ubuntu)
          # -X POST = send an HTTP POST request
          # The URL is the Render deploy hook (stored as a GitHub secret)
          # When Render receives this POST, it immediately starts a new deployment:
          #   1. Pulls the latest commit from GitHub
          #   2. Runs 'npm install'
          #   3. Runs 'node server.js'
          #   4. Replaces the old instance with the new one
          # ${{ secrets.RENDER_DEPLOY_HOOK_URL }} = reads the GitHub Actions secret.
          # Secrets are encrypted in GitHub and never appear in logs.

Full Pipeline Flow

You edit code in VS Code ↓ git add . → git commit -m "..." ↓ git push origin main ↓ (GitHub receives the push) GitHub Actions detects push to main branch → Allocates Ubuntu VM → Runs deploy.yml → Executes: curl -X POST "${{ secrets.RENDER_DEPLOY_HOOK_URL }}" ↓ Render receives the POST on its deploy hook endpoint → Pulls latest commit from GitHub repo → Runs: npm install → Runs: node server.js → Health check passes → Swaps traffic to new instance ↓ ~2 minutes after git push → new code is live at https://project3-music-app.onrender.com
